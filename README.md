# The Agentic Players Ball

A private circle for the founders, builders, and researchers shaping the agentic era.

Live at [agenticplayersball.com](https://agenticplayersball.com).

---

## What this is

A single-page invite-application site for The Agentic Players Ball, available in **seven languages** (English, German, French, Spanish, Brazilian Portuguese, Mandarin Chinese, Japanese). Visitors fill out a form requesting an invite; submissions land in Airtable as **Applicant** records for review.

This README is the technical reference for the project — how it fits together, how to make changes, and the known gotchas.

## i18n / multi-language

Source of truth: **`i18n/<lang>.json`** (one file per language). Each file has the same 45 keys. Edit a translation, push, and Cloudflare rebuilds.

### URL structure

```
agenticplayersball.com/         → EN  (default)
agenticplayersball.com/de/      → DE
agenticplayersball.com/fr/      → FR
agenticplayersball.com/es/      → ES
agenticplayersball.com/pt/      → PT-BR
agenticplayersball.com/zh/      → ZH-Hans
agenticplayersball.com/ja/      → JA
```

Each variant is a real static HTML file with translations baked in at build time — search crawlers see fully localised content on first byte. `?lang=de` legacy URLs redirect to `/de/` via a small JS stub.

### Language detection

On first visit, the page is selected by, in order:
1. URL param `?lang=XX` (legacy compat, redirects to `/<XX>/`)
2. `localStorage['apb-lang']` (persisted from previous switcher click)
3. `navigator.language` matched against supported prefixes (`zh`, `ja`, `de`, `fr`, `es`, `pt`)
4. Falls through to EN

User clicks on the footer language switcher navigate to the clean URL and persist the choice to localStorage.

### Translation conventions baked into the locale files

- The brand name `"The Agentic Players Ball"` is **never translated** — it appears verbatim in `meta.title`, `success.body1`, and `footer.copyright` of every locale.
- The mascot name `"Sparkles"` is preserved (transliterated to katakana in JA only).
- Identity chips (Founder / Builder / Researcher) are **not** translated — they're written into Airtable as English values, so the labels stay English in every locale.
- The word `"builders"` stays in English in ES and PT (deliberate brand choice).
- `intro.sentence1` is **HTML** — it includes a `<span class="accent">…</span>` highlight wrapping the grammatically appropriate phrase per language. Japanese places the modifier clause before the noun (`<span class="accent">エージェントの時代を形作る</span>ファウンダー…`) — that's correct Japanese grammar, not a bug.
- Chinese uses six-dot ellipsis (`……`, two U+2026); other languages use single ellipsis (`…`).
- French uses typographic apostrophes (`'`, U+2019). Spanish uses inverted `¿` `¡`. ZH and JA use full-width punctuation (`，` `。` `？`).

### How the build wires it up

`build-i18n.js` opens the source `index.html`, then for each locale:

1. Sets `<html lang>` and adds `lang-XX` class on `<html>` for CSS scoping.
2. Replaces `<title>` and updates `meta[name=description]`, `og:description`, `og:locale`, `og:url`, `twitter:description`.
3. Updates `<link rel="canonical">` to the variant URL.
4. Strips existing `<link rel="alternate" hreflang>` tags and re-emits a fresh block with all 7 + `x-default`.
5. Updates Schema.org `Organization.description` + `WebSite.inLanguage` and appends a per-page `WebPage` block with `isPartOf` linking to the WebSite.
6. Walks every `[data-i18n]` (textContent), `[data-i18n-html]` (innerHTML — used for `intro.sentence1` only), `[data-i18n-placeholder]`, and `[data-i18n-aria]` and substitutes from the locale JSON.
7. Sets the phrase animation's `data-text` and `aria-label` to the translated phrase so the letter-split animation reads the right language.
8. Marks the active language pill in the footer with `class="active"`.
9. Replaces the runtime i18n engine in `<head>` with a small stub that exposes `window.__i18nData` (so the form's `t()` lookups for error messages work), fires `i18nReady` on DOMContentLoaded (so the phrase animation runs), persists to localStorage, and redirects legacy `?lang=` URLs.

After all 7 are written, the script generates `dist/sitemap.xml` (one entry per URL with full hreflang block) and `dist/robots.txt` (allow all, disallow `/api/` + `/functions/`, point at the sitemap).

## How the form works

The form (`<form id="signup-form">` in `index.html`) is plain HTML enhanced with vanilla JS:

1. **Field validation runs on the client first** — touch-on-blur, visual error states (`.field.is-invalid`), per-field red error messages (`<small class="field-error">`), submission is blocked client-side if any field is invalid. The first invalid field scrolls into view. Error messages read from `window.__i18nData` so they're in the visitor's language.

2. **On submit**, the JS POSTs JSON to `/api/signup` (the Pages Function). Submit button shows the localised "Submitting…" while in flight.

3. **The Pages Function re-validates everything server-side** (never trust the client) and POSTs to Airtable. If Airtable accepts the record, the function returns `{ ok: true }`. Otherwise it returns a human-readable error.

4. **On success**, the entire signup card is replaced with a localised thank-you message. On failure, a red error banner appears below the submit button.

### Validation rules (client and server are kept in sync)

| Field | Required | Rule |
|---|---|---|
| First name | yes | non-empty (auto title-cased server-side) |
| Last name | yes | non-empty (auto title-cased server-side) |
| Email | yes | valid format AND not a disposable domain (mailinator, tempmail, etc.) |
| LinkedIn | yes | matches `linkedin.com/in/`, `/company/`, or `/pub/` (with or without `https://`) |
| GitHub | no | if filled, must match `github.com/...` |
| Telegram Handle | yes | public username only — `@handle`, `handle`, `t.me/handle`, or `https://t.me/handle` all accepted, normalised to `@handle` |
| Identity | yes | at least one of Founder / Builder / Researcher |
| Why are you agentic? | yes | non-empty |
| Have you been referred? | no | if checked, referrer LinkedIn URL is required and must validate as LinkedIn |

Server-side normalisation:
- Names → Title Case (with apostrophe + hyphen handling: `o'brien-jones` → `O'Brien-Jones`)
- Email → lowercased
- URLs without `https://` → prepended automatically
- Trailing slashes stripped
- Telegram → always saved as `@handle`

## Adding a new language

1. **Translate** — copy `i18n/en.json` to `i18n/<code>.json`, translate all 45 keys. Match the conventions of the existing locales for that script (typographic apostrophes for European, full-width punctuation for CJK, etc.).
2. **Wire it into the build** — add the code to the `LOCALES` array and the `HREFLANG` map in `build-i18n.js`.
3. **Add a footer pill** — add a `<button class="lang-btn" data-lang="<code>">…</button>` plus a `<span class="sep">|</span>` separator inside `<nav class="lang-switcher">` in `index.html`. Keep CJK distributed (don't cluster Asian languages adjacent).
4. **Wire detection** — add a `n.startsWith('<code>')` check in the head boot script's `detectLang()` and add the code to the `SUPPORTED` array (in two places — both the head boot and the stub it gets replaced with in `generateStub()`).
5. **Update the spec table at the top of this README** if you want to be tidy.
6. Push. CF rebuilds with the new locale. New URL `agenticplayersball.com/<code>/` is live, hreflang updates everywhere.

## Known gotchas

- **Image/asset paths must be absolute.** `<img src="agenticplayers.png">` works at `/` but 404s at `/de/agenticplayers.png` because Cloudflare's SPA fallback then returns HTML instead of a PNG and the image breaks. Always use `/agenticplayers.png` (leading slash). Same rule for any other asset referenced by relative path.

- **The phrase animation splits each letter into `display:inline-block`,** which means without intervention the browser will happily wrap mid-word. The CSS rule `html:not(.lang-zh):not(.lang-ja) .phrase .word { white-space: nowrap }` keeps each Latin word together. CJK is excluded because there's no whitespace between characters and wrapping mid-character is the natural behaviour.

- **Cloudflare config changes don't retroactively rebuild.** Editing the Build command or Output directory only affects the next deploy. Push a commit (or click Retry deployment) to apply.

- **Cloudflare env vars need a redeploy** to take effect. Adding/changing them silently doesn't update running functions.

- **Pages Functions don't run locally** with a plain HTTP server. Use Cloudflare preview branches for end-to-end testing.

- **Safari caches favicons aggressively** — even with `?v=N` cache-busts. To force-refresh: Settings → Privacy → Manage Website Data → search the domain → Remove. Then quit and reopen Safari.

- **MEGA sync ≠ git ignore.** `.secrets/` is gitignored but MEGA still syncs it across devices and to their cloud. Production secrets must live in Cloudflare env vars only.

- **The disposable email blocklist is hardcoded** in `functions/api/signup.js`. ~30 of the most common throwaway domains. If a determined applicant uses a niche disposable provider, they get through. The blocklist is best-effort, not bulletproof.

- **The mascot has a face that disappears at favicon sizes.** That's why the favicon is a separate clean disco ball (no face) — the SVG and the rasterised PNGs/ICO all use the simplified version.

- **Form chip selection state uses CSS `:has()`.** Works in all modern browsers (Safari 15.4+, Chrome 105+, Firefox 121+). If you need to support older browsers, this needs a JS fallback.

- **Identity chip values stay English in Airtable.** Even when a Spanish-speaking applicant submits the form, the Identity field is `["Founder", "Builder"]` not `["Fundador", "Builder"]`. This is by design so reviewing applicants in Airtable doesn't require knowing seven languages.

- **The build runs cheerio against the source HTML.** If you add a new `<script>` block in `<head>` that happens to contain the words `SUPPORTED`, `detectLang`, AND `applyTranslations`, the build will mistakenly replace it with the i18n stub. Vanishingly unlikely but worth knowing.

## Adding a new field to the form

1. Add the input to `index.html` inside `<form id="signup-form">`. Use `data-i18n="form.<key>"` on the label so it gets localised.
2. Add a `<small class="field-error"></small>` slot beneath it.
3. Add the new key + every error message string to **all 7** `i18n/*.json` files. The form's `t()` function pulls error strings from `window.__i18nData`, so missing keys = English fallback.
4. Add a validation rule to `getRules()` near the bottom of `index.html` (or to `validateReferrer` / `validateIdentity` for special cases).
5. Add the field to the JSON payload in the submit handler.
6. In `functions/api/signup.js`: parse the field, validate it, add it to the `fields` object passed to Airtable.
7. In Airtable: create the field on the People table.
8. Push, CF builds, test.

## When something breaks

- **`/de/` (or any locale URL) returns 404 / serves source HTML** — the build didn't run on Cloudflare. Check Settings → Builds & deployments has `Build command: npm run build`, `Output directory: dist`. Then push any commit (or click Retry deployment) to trigger a rebuild.
- **All locale URLs return identical EN content** — same diagnosis: build never ran, Cloudflare is serving the source `index.html` from the repo root with SPA fallback.
- **`/de/` shows EN content despite build succeeding** — i18n data isn't loading. Check `<html lang>` in served HTML; if it says `en`, the build script's translation step didn't run for that locale. Check `i18n/de.json` is valid JSON (run `python3 -c "import json; json.load(open('i18n/de.json'))"`).
- **Image (disco ball) is broken on `/de/` but works on `/`** — the `<img src>` is relative. It must be absolute (`/agenticplayers.png`). See gotcha above.
- **Phrase animation breaks "Nous" mid-word in French** — `.phrase .word` is missing the `white-space: nowrap` rule. Already fixed but if it regresses, this is the cause.
- **Form returns 500** — check Cloudflare Pages → Functions → Real-time logs. Most likely an env var is missing or the Airtable schema doesn't match.
- **Form returns 502** — Airtable rejected the record. Likely a field name typo or a value type mismatch (e.g. sending a string to a multipleSelects field).
- **Form error messages show in English on a non-EN page** — the locale JSON is missing a `form.error.*` key, so `t()` falls back. Add the key.
- **No record in Airtable but form shows success** — would be a bug in the Pages Function, but currently it returns `ok: true` only after Airtable confirms the insert. Check the Airtable view filter — your record might be hidden by a filter rather than missing.
- **Favicon won't update** — Safari cache. See gotcha above.
- **Form looks great on desktop but broken on mobile** — check `@media (max-width: 700px)` block in `index.html`. That's where mobile overrides live.
